# 需求文档

## 简介

AI 编程 Agent 可以快速完成需求分析、技术设计、任务拆分和代码实现，但同一 Agent 往往会沿用自己的假设，难以发现方案偏离、错误架构落点、范围膨胀和遗漏约束。长任务或新会话还可能丢失只存在于聊天中的限制。

Intent Review Engine 保存任务的原始意图和已确认决策，并使用全新、只读的 Reviewer 独立审查方案与最终实现。核心引擎不绑定具体宿主或模型；首版通过 Codex 插件提供交互入口。

## 角色

- **用户**：提出需求、补充约束并决定是否接受审查发现。
- **Worker**：当前负责生成方案和实现代码的 Agent。
- **Reviewer**：使用全新上下文、只读访问仓库的独立 Agent。
- **Intent Review Engine**：维护 Task、证据、审查协议、预算和决策记录的本地核心。
- **Host Adapter**：把 Codex、Claude Code 等宿主操作转换为统一引擎调用。

## 需求

### 需求 1 - 建立任务意图契约

**用户故事：** 作为使用 AI Agent 开发的用户，我想把原始需求和后续约束保存为持久文件，以便长任务和新会话不会遗忘关键意图。

#### 验收标准

1. 当用户开始一个新任务并调用初始化能力时，系统应该创建唯一任务目录并保存用户原始需求、目标、非目标、明确约束、禁止项和待确认假设。
2. 当原始需求可从当前上下文获得时，系统应该优先保留用户原文，而不是只保存 Agent 的概括。
3. 当用户补充、修改或否决某项约束时，系统应该追加带时间和来源的决策记录，不得静默覆盖历史内容。
4. 当新会话接手已有任务时，系统应该只依赖任务目录即可恢复当前有效约束、批准状态和未解决问题。
5. 如果无法可靠获得原始用户表达，那么系统应该明确标记证据缺失，不得假装已经完整捕获需求。
6. 每个任务应该拥有独立于 Session 的稳定 Task ID，并记录关联仓库、分支、阶段和参与过的 Session。
7. 当新会话所在仓库与当前分支只有一个活跃 Task 时，系统应该自动恢复该 Task 并明确告知用户；当同一仓库与分支存在多个活跃 Task 时，系统应该列出候选并要求用户选择，不得猜测。
8. 当用户恢复 Task 时，系统应该提供目标、有效约束、已批准方案、未解决发现、当前阶段和 Git 基线的简洁交接摘要。
9. 当用户在方案批准后变更契约时，系统应该追加决策记录、把当前方案快照标记为过期并要求重新审查受影响部分，不得继续以过期快照作为实现审查基准。

### 需求 2 - 独立审查方案文档

**用户故事：** 作为方案审批者，我想让一个没有参与方案生成的 Reviewer 对照原始意图和仓库事实检查文档，以便在编码前发现方向性错误。

#### 验收标准

1. 当用户请求方案审查时，系统应该启动新的只读 Reviewer 上下文，不得传递 Worker 的隐藏推理或自我评价。
2. Reviewer 应该首先检查 `contract.md` 是否忠实于 `source.md`，包括约束是否被遗漏、被降级为假设或被扩写出原文不存在的内容。
3. Reviewer 应该检查需求遗漏、无依据扩展、错误架构层、模块耦合、过大改动范围、不可验证的验收标准和缺失风险。
4. 每条审查发现都应该包含严重程度、类别、具体主张、来源证据、影响、建议和置信度。
5. 当发现仅属于偏好或可选优化时，Reviewer 应该将其标记为建议，不得作为阻断问题。
6. 当审查证据不足时，Reviewer 应该返回“无法验证”，不得臆测为通过或失败。
7. 当 `contract.md` 存在未解决的阻断级失真时，系统不得允许方案进入批准状态。
8. 当用户明确批准方案时，系统应该保存不可变的方案快照、对应 Git 基线和批准时间。

### 需求 3 - 审查最终实现的一致性

**用户故事：** 作为需求负责人，我想在实现完成后核对代码是否真的满足需求并遵守已批准方案，以便在提交前发现实现跑偏。

#### 验收标准

1. 当用户请求实现审查时，系统应该比较原始意图、有效约束、已批准方案和从批准基线开始的实际改动。
2. 系统应该为每条验收标准给出“已实现、部分实现、缺失或无法验证”的状态和证据。
3. 系统应该列出每个修改文件对应的需求或任务、修改理由以及范围合理性判断。
4. 当实现增加方案未声明的公共接口、依赖、数据库变更、跨模块耦合或底层改动时，系统应该明确标记方案偏离。
5. 当测试通过但没有证明目标行为时，系统不应该仅根据绿色测试判定需求完成。
6. 当工作区包含无法归属于当前任务的改动时，系统应该把它们标记为范围不明，不得自动纳入通过结论。

### 需求 4 - 审查发现的裁决与留痕

**用户故事：** 作为用户，我想逐条决定如何处理 Reviewer 的发现，以便避免 Agent 自动接受错误建议。

#### 验收标准

1. 当 Reviewer 返回发现时，Worker 应该先核验证据，再向用户呈现结论。
2. 用户应该能够将发现标记为接受、拒绝、延期或已解决，并保存理由。
3. 系统不得在没有用户明确授权时自动修改方案或源代码。
4. 当再次审查时，Reviewer 应该获得此前发现和裁决，以便确认问题是否解决且避免重复争论。
5. 当仍有阻断发现未解决时，系统不得输出“审查通过”。

### 需求 5 - 宿主与 Reviewer 可替换性

**用户故事：** 作为使用者，我想从不同 Agent 宿主调用同一任务审查引擎，并选择同模型新会话或不同模型进行审查，以便兼顾可用性、成本和独立性。

#### 验收标准

1. 核心 Task、证据和决策能力应该由本地 Engine 提供，不应该依赖 Codex Skill 的聊天上下文保存状态。
2. 首版应该提供 Codex Host Adapter 和 Codex 全新只读 Reviewer。
3. Host Adapter 应该只处理宿主交互、Reviewer 启动和结果展示，不应该重复实现 Task 状态逻辑。
4. 审查输入和输出应该使用与具体宿主和模型无关的数据结构。
5. 当未来接入 Claude Code、CLI 或其他 Reviewer 时，不应该改变任务目录和审查报告格式。
6. 当 Reviewer 不可用、超时或返回格式错误时，系统应该记录失败状态，不得把失败当作通过。

### 需求 6 - 安全和范围控制

**用户故事：** 作为代码所有者，我想确保审查过程不会修改代码或泄露无关信息，以便安全地在真实项目中使用。

#### 验收标准

1. Reviewer 应该默认运行在只读沙盒中。
2. 首版不应该自行访问网络或向当前已授权的 Codex 宿主之外发送仓库内容。
3. 系统应该限制证据包为任务相关文档、仓库规则、相关代码和 Git Diff。
4. 当 Evidence Builder 发现仓库业务文件时，系统应该遵循仓库忽略规则、路径边界和敏感信息过滤。
5. 当系统读取当前 Task 的 `source.md`、`contract.md`、决策记录、方案快照和历史发现时，仓库的 `.gitignore`、`.ignore` 或默认扫描排除规则不应该过滤这些任务证据。
6. 当任务证据或仓库证据可能包含密钥或敏感信息时，系统应该停止向 Reviewer 发送可疑内容；任务证据绕过仓库忽略规则不得绕过安全过滤。
7. Task Store 和审查运行产物默认不应该出现在业务 Git Diff 与文件范围矩阵中。

### 需求 7 - 成本预算与分层审查

**用户故事：** 作为频繁使用 AI Agent 的用户，我想限制每个任务的审查成本，以便在保持关键节点质量的同时避免过量 Token 消耗。

#### 验收标准

1. 首版应该默认使用 Balanced 成本模式，只在方案完成后和实现完成后调用强 Reviewer。
2. Engine 应该记录每轮审查的输入规模、Reviewer、调用次数和可获得的 Token 用量。
3. Engine 应该支持单次输入上限、任务总预算和最大审查轮数；超出预算时应该请求用户确认，而不是自动继续。
4. Evidence Builder 应该使用 Git 元数据、文件哈希和增量 Diff 避免重复发送未变化内容。
5. 当本地或低成本 Reviewer 被启用时，系统应该允许它进行结构化初审，并把高风险、低置信度或复杂问题升级给强 Reviewer。
6. 本地或低成本 Reviewer 不得单独为高风险任务给出最终通过结论，除非其能力已通过项目 Fixture 验证并由用户显式配置。

### 需求 8 - 可审计的 Reviewer 质量闸门

**用户故事：** 作为产品开发者，我想在实现 Engine 前用预先登记的标准评估 Reviewer，以便避免根据实验结果事后调整成功定义。

#### 验收标准

1. 当首次执行 Reviewer 质量评估前，项目应该登记目标缺陷召回率、高严重度误报率、证据有效率和重复运行稳定性的通过阈值。
2. 当建立基线时，基线应该只用于确认 Fixture 中的目标缺陷、对照样本和标签是否有效，不得用于反推或放宽通过阈值。
3. 当基线证明 Fixture 或预注册阈值存在明确设计问题时，项目最多可以调整一次阈值，并在下一轮评估前记录原值、新值、原因、证据和用户批准。
4. 当第二轮评估仍未达到预注册或唯一一次调整后的阈值时，项目应该修订审查协议或产品假设，不得继续调整阈值使结果通过。
5. 当评估 Reviewer 时，评估集应该覆盖多个目标缺陷类型，并包含没有目标缺陷的对照 Fixture 和重复运行结果。

## 非功能需求

- 首版采用本地文件存储，不依赖数据库或云端服务。
- 首版命令应在小型和中型 Git 仓库中可用。
- 审查报告应便于人类阅读，同时保留稳定的结构化字段供后续自动化使用。
- 核心流程不应依赖特定的 requirements/design/tasks 文件命名；默认支持这些名称，并允许用户指定路径。
- 核心引擎应提供可被不同 Host Adapter 调用的稳定 CLI 或等价本地接口。
- 核心引擎的分发方式应该明确运行时依赖，不得把必要依赖推迟到发布阶段才暴露。
- 核心流程应该在 Windows、macOS 和 Linux 工作，并正确处理包含空格、中文和特殊字符的仓库路径。

## 首版范围外

- 自动修复和自动提交。
- 多 Reviewer 投票与自动共识循环。
- GitHub PR Bot 和远程托管服务。
- 自动阻断 Worker 的每一次文件修改。
- Agent 执行过程的持续监控、自动提醒和暂停。
- 非 Git 项目的完整实现审查。
