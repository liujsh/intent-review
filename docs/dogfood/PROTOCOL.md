# 手动 Dogfood 协议

目的不是「做好审查」，而是**采集决定引擎该不该建、该建多小的证据**。

唯一的核心指标：**有没有哪条发现真的改变了我接下来的动作。**

零代码。全部手动。每个任务额外开销约 30–45 分钟。

---

## 任务选择

跑 3 个，配比固定：

| # | 类型 | 目的 | 选取标准 |
| --- | --- | --- | --- |
| 1 | 回溯任务 | 测召回。已知答案，作为对照 | 你**已经踩过坑**的历史任务：当时方案有问题，后来才发现。用当时的原始需求和当时的方案文档，不要用修好之后的版本 |
| 2 | 真实任务 | 测价值。未知答案 | 你接下来本来就要做的事 |
| 3 | 真实任务 | 测稳定性。新鲜感褪去后还成不成立 | 同上，与 #2 不同类型 |

任务必须满足：多步、碰得到架构、有容易在长对话里丢掉的约束。琐碎任务测不出漂移，浪费时间。

#1 直接对应 design 第 10 节的第一条成功指标（「在人工已经发现问题的历史方案中，Reviewer 能否提前发现同类问题」），且因为你已知答案，它是唯一能算召回率的样本。先跑它，最便宜。

---

## 目录

所有证据集中放在本仓库，不污染被审查的目标仓库（顺带绕开 PLAN-004）：

```text
docs/dogfood/<task-slug>/
  source.md        # 你自己写，逐字
  contract.md      # Worker 生成
  plan/            # Worker 产出的方案文档
  snapshot/        # 批准时 plan/ 的副本
  review-r1.md     # Reviewer 输出
  decisions.md     # 你的裁决
  notes.md         # 采集表（真正的产出）
```

## 步骤

### 0. 写 source.md —— 这步不能委托

在任何 Agent 碰这个任务**之前**，你自己把原始需求逐字写下来。

这是整个实验唯一不可省的一步。让 Agent 代写等于污染对照组——你就再也无法判断 contract 的失真是 Worker 引入的，还是本来就没说清楚。

规则：写完只许追加，不许改。后续补充的约束追加到文件末尾，带时间。

### 1. Worker 正常工作

让 Codex（或 Claude）按你平时的习惯产出方案文档，放进 `plan/`。不要为了实验改变你的正常做法。

### 2. Worker 生成 contract.md

从 `source.md` 提炼：目标、非目标、约束、禁止项、待确认假设。

### 3. 换一个全新上下文的 Reviewer

**默认做法**：开一个新会话（Codex 或 Claude，与 Worker 不同的那个更好），只给它：

- `source.md`
- `contract.md`
- `plan/` 下的方案文档
- 仓库规则（`AGENTS.md` / `CLAUDE.md`）
- 目标仓库的只读访问

**不给**：Worker 的聊天记录、它的自我评价、你的怀疑。

> 想顺手把 PLAN-001 的 Spike 一起做掉，就改用 `codex exec`——它默认只读，配 `--output-schema` 能拿结构化输出，`--json` 里带 token usage。一条命令同时完成 dogfood 和 Runtime 验证。但如果这会拖慢你开始，就先用新会话，Spike 单独做。

Reviewer 的指令要点（照抄即可）：

```text
你是独立 Reviewer，未参与方案生成，只读。

审查顺序：
1. contract.md 是否忠实于 source.md：约束是否遗漏、是否被降级为假设、是否扩写出原文没有的要求。
2. source.md 的每条诉求是否进入方案和验收标准。
3. 方案是否引入无依据的范围、公共抽象或额外依赖。
4. 改动是否落在正确的模块和架构层。
5. 任务和测试是否足以证明需求完成。

每条发现必须包含：
severity(blocker/high/medium/advisory)、category、claim、
evidence(文件+行号+说明，至少一条)、impact、recommendation、confidence

规则：
- 没有仓库或需求证据的风格偏好，不得升级为 blocker 或 high。
- 证据不足时返回「无法验证」，不要臆测通过或失败。
```

### 4. 逐条裁决，当场写

每条发现在 `decisions.md` 记一行：`accepted / rejected / deferred / resolved` + 理由。

**当场写，不要事后回来合理化。** 事后补的裁决全是马后炮，采集不到真实信号。

### 5. 冻结快照

批准时把 `plan/` 复制成 `snapshot/`，记下当时的 `git rev-parse HEAD`。

### 6. 实现，然后第二次审查

再开新会话，给 `source.md` + `contract.md` + `snapshot/` + `git diff <baseline>..HEAD`，产出两张矩阵：

```text
验收标准 → 实现位置 → 测试/运行证据 → implemented/partial/missing/unverifiable
修改文件 → 对应需求  → 修改理由      → expected/suspicious/out-of-scope
```

## 判读

3 个任务跑完看两件事：

**动作变更率。** 三个任务加起来，如果被改变动作的发现少于 3 条，或者全部集中在 #1（回溯任务，你本来就知道答案），那么价值主要来自新鲜感，引擎不值得建。

**手动重复劳动清单。** 采集表第 3 节列出来的东西，就是最小闭环该做的全部。**没进这个清单的功能，首版都不要做**——包括你现在方案里已经写好的那些。

## 反作弊

你既是被审的人也是判卷的人，以下几条是防自己的：

- 不要告诉 Reviewer 你怀疑哪里有问题。
- 不要因为想让实验成功而选一个你已经知道有毛病的活任务（#1 除外，它的作用就是已知答案）。
- 「我本来也会发现」这句话，只有在你能指出**已经写在哪里**时才算数，否则记「会但要晚很久」。
- 三个任务跑完之前不许改协议。要改，记在 `notes.md` 里下一轮再改。
